# План переписывания вычислительного ядра COMTAILS (этап 1: расчёты)

## Цель этапа

Перенести **только численно тяжёлую часть** COMTAILS с Python на высокопроизводительный стек, сохранив физический смысл модели и воспроизводимость результатов.

Критерии успеха этапа:
- ускорение не менее чем в 10–30x на CPU/GPU-сценариях реальной задачи;
- численная сходимость с текущей Python-версией (допустимые отклонения задаются тестами);
- модульная архитектура, чтобы позже безопасно подключить GUI на Rust и веб-слой.

---

## Что сейчас является узкими местами

По коду видно, что основная стоимость времени уходит в:

1. Вложенные циклы по времени, размерам частиц и событиям выброса в `DustTail.build`.
2. Повторные вызовы орбитального расчёта для каждой частицы (`heliorbit`).
3. Пиксельные накопления потока и оптической толщины.
4. Ручная свёртка изображения тройными циклами в `apply_convolution`.

Это типичный профиль задачи для ускорения через C++ + CUDA: много независимых однотипных вычислений, регулярные численные операции, большой объём параллельной работы.

---

## Выбор технологии: C++/CUDA vs Fortran + CUDA

### Рекомендация: **C++ + CUDA**

Почему это лучше для вашего плана:

1. **Лучшая интеграция с будущим Rust GUI**
   - C ABI и FFI между Rust и C/C++ проще и популярнее в современной инфраструктуре.

2. **Экосистема и инструменты**
   - Удобнее профилировать и оптимизировать через Nsight Systems / Nsight Compute.
   - Проще найти готовые паттерны для memory coalescing, shared memory, reduction, atomics.

3. **Гибрид CPU+GPU в одном коде**
   - Можно держать единый backend: CPU-реализация на C++ и ускоренная CUDA-ветка.

4. **Поддержка и найм/развитие проекта**
   - Для долгосрочного open-source/production проекта найти инженеров C++/CUDA обычно проще.

### Когда Fortran + CUDA мог бы быть лучше

- Если команда уже глубоко Fortran-ориентирована и есть большой legacy-код именно на Fortran, который нужно переносить почти без изменений.

В вашем случае текущая база — Python, а следующий этап — Rust GUI, поэтому **C++ + CUDA более целевой и практичный выбор**.

## Предлагаемая архитектура вычислительного ядра

```text
comtails-core/
  include/
    comtails_core.h          # C ABI для вызова из Python/Rust
  src/
    core_cpu.cpp             # CPU fallback
    core_cuda.cu             # GPU kernels
    orbit_math.cpp/.cu       # орбитальная математика
    image_accumulation.cu    # накопление flux/opt_depth
    convolution.cu           # GPU свёртка
  tests/
    reference_regression.cpp
```

### API первого этапа

Один входной вызов (batched):
- вход: массивы состояния кометы/параметров выброса/сетка/параметры камеры;
- выход: `flux`, `opt_depth`, `opt_depth_nuc`, агрегированные скалярные метрики.

Важно: API должен быть **детерминированным** по seed генератора случайных чисел.

## Стратегия миграции (без риска)

### Шаг 1. Базовый профайлинг Python (фиксируем baseline)
- Собрать 3 эталонных сценария: small / medium / large.
- Замерить wall time, CPU time, использование RAM, если возможно — распределение по функциям.
- Сохранить эталонные выходы (`tail_sdu`, `OPT_DEPTH`, `afrho`).

### Шаг 2. C++ CPU-референс (без CUDA)
- Перенести формулы 1:1 с Python, без «умных» оптимизаций.
- Проверить отклонения от Python по метрикам:
  - max/mean relative error для `flux` и `opt_depth`;
  - абсолютная ошибка для `afrho`/`mag`.

### Шаг 3. CUDA для самого тяжёлого ядра
- Параллелизовать генерацию/трассировку частиц и накопление в изображение.
- Использовать SoA-раскладку данных (structure-of-arrays) для coalesced memory access.
- Снизить contention на атомиках через блочную агрегацию (shared memory + merge).

### Шаг 4. CUDA свёртка изображения
- Заменить текущую ручную свёртку на separable convolution kernel.
- Для больших кадров рассмотреть FFT-подход, но сначала separable (он проще и обычно достаточен).

### Шаг 5. Интеграционный слой
- Поднять Python binding (pybind11) для поэтапной подмены `DustTail.build`.
- Подготовить стабильный C ABI, который затем будет вызываться из Rust.

## Точность и воспроизводимость

Чтобы ускорение не сломало физику модели:

1. Зафиксировать режимы вычислений:
   - основной режим: `float64` для научной воспроизводимости;
   - быстрый режим: смешанная точность (где безопасно).

2. Детеминированный RNG:
   - фиксированный seed;
   - независимые подпоследовательности на поток/блок.

3. Регрессионные тесты:
   - сравнение массивов по tolerances;
   - проверка инвариантов (неотрицательность яркости, физические ограничения).

---

## Минимальный план работ на ближайший спринт

1. Сформировать benchmark-набор входов и эталонные выходы.
2. Выделить из Python чёткий контракт вход/выход для вычислительного ядра.
3. Создать `comtails-core` с C++ CPU-версией и тестами эквивалентности.
4. Перенести главный Monte Carlo-цикл на CUDA.
5. Подключить ядро обратно в Python через pybind11 и сравнить end-to-end.

---

## Риски и как их закрыть

- **Расхождение чисел между CPU/GPU** → заранее прописать tolerances и deterministic mode.
- **Узкое место на атомиках при накоплении в пиксели** → двухфазная агрегация и тайлинг.
- **Сложность отладки CUDA** → сначала CPU parity, затем kernel-by-kernel перенос.
- **Слишком ранняя оптимизация** → сначала correctness, потом Nsight-driven tuning.

## Итог решения

Для пункта «Расчёты» выбираем **C++ + CUDA** как основной путь.

Это наиболее рационально для:
- максимального ускорения текущей вычислительной части;
- чистой интеграции с вашим будущим стеком Rust GUI;
- дальнейшего расширения до Docker и веб-демо без архитектурного долга.
